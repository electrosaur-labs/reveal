# Reveal Recipe Engine — Design Document

## Motivation

With Wu quantization working and archetypes maturing, we need:

1. **Dynamic archetype parameters** — expressions like `channels + 2` instead of static values
2. **Record/replay** — capture a separation session as a reproducible artifact
3. **Portable recipes** — apply the same separation to similar images

The scripting format is plain JavaScript. A recipe is a function `(image, engine) => result` that calls methods on a well-defined API. No custom parser, no grammar — JS gives us arithmetic, conditionals, and loops for free.

---

## Design Principles

**Four types, clear ownership.** The API has exactly four types: Image (immutable input), Engine (reusable configuration), Palette (mutable working set), and Result (output with post-processing). Each has a single responsibility.

**Fail loud on bad input.** Every method validates its arguments and throws on unrecognized keys, out-of-range values, wrong types, or misspelled parameter names. No silent no-ops. A typo like `targetColrs` should produce an immediate, clear error — not a mysterious default behavior three steps later. No try/catch inside the engine — error handling is the recipe author's responsibility.

**Host-agnostic engine.** Photoshop (or any host) is just a source and sink of image data. The four types are pure computation with no host dependencies. This guarantees the same engine can power a CLI, a TUI/REPL, a desktop GUI, or a Photoshop plugin.

**Image is frozen per recipe execution.** The host guarantees the image doesn't change while a recipe runs. In Navigator, "Reinject" creates a new Image and starts a new recipe execution — it does not mutate the current one.

**No hidden state.** `quantize()` and `separate()` take the Image as an explicit argument. The Engine doesn't hold a reference to it. Data flows through function arguments, not side effects.

## The Four Types

### Image — Immutable Input

Created by the host from pixels. Frozen for the lifetime of a recipe. Contains pixel data + computed DNA. The recipe receives it as a read-only argument.

```javascript
image = {
    width: 4000,
    height: 3000,
    bitDepth: 16,
    colorSpace: "Lab",
    labPixels: Uint16Array,         // raw pixel data (host-provided)
    dna: { /* full DNA v2.0 */ },
    sectors: { /* 12 hue sectors */ },
    filename: "ayyanar-horse.psd"   // optional — unavailable when ingesting from Photoshop
}
```

### Engine — Reusable Configuration

Accumulates archetype and parameter settings. Stateless — does not hold image data or palette. Can be reused across multiple images.

```javascript
const engine = new Engine();
engine.applyArchetype("salamander", { mesh: 355 });
engine.setParam("lWeight", 1.5);

// Same engine, different images
const palette1 = engine.quantize(horse, { targetColors: 8 });
const palette2 = engine.quantize(landscape, { targetColors: 6 });
```

### Palette — Mutable Working Set

Produced by `engine.quantize()`. The only mutable object in the pipeline. Supports surgery operations (merge, remove, add, edit). Passed explicitly to `engine.separate()`.

### Result — Output + Post-Processing

Produced by `engine.separate()`. Contains the color index map, palette snapshot, and masks. Mechanical knobs (speckleRescue, shadowClamp, minVolume) operate on the Result without re-separating — just like Navigator does today for real-time knob tweaking.

## Recipe Format

```javascript
// Recipe: Ayyanar Horse #1
// Generated by Reveal Navigator 2026-03-11
// Archetype: salamander | Image DNA: { l:52, c:28, k:71, ... }

module.exports = (image, engine) => {
    // Phase 1: Configure
    engine.applyArchetype("salamander", { mesh: 355 });

    // Phase 2: Quantize — produces Palette
    const palette = engine.quantize(image, { quantizer: "wu", targetColors: 8 });

    // Phase 3: Palette Surgery
    palette.merge("#BBD900", "#FFBC4E");
    palette.find("#000000").setWeight(2.5);

    // Phase 4: Separate — produces Result
    const result = engine.separate(image, palette);

    // Phase 5: Mechanical Knobs — mutates Result
    result.applyKnobs({ speckleRescue: 2, shadowClamp: 3.2, minVolume: 0.5 });

    return result;
};
```

## Engine API

### Configuration

#### `engine.applyArchetype(name, overrides?)`

Loads a named archetype and applies its parameters. Optional overrides merge on top. Dynamic expressions in the archetype are evaluated against the image context at `quantize()` time (see Dynamic Expressions below).

```javascript
engine.applyArchetype("salamander");
engine.applyArchetype("salamander", { mesh: 355, targetColors: 10 });
```

Throws if archetype name is unknown.

#### `engine.setParam(key, value)`

Set any structural/mechanical/production parameter individually. Throws on unrecognized keys.

```javascript
engine.setParam("lWeight", 1.5);
engine.setParam("vibrancyMode", "exponential");
engine.setParam("distanceMetric", "cie94");
engine.setParam("preprocessingIntensity", "light");
```

Valid keys: `targetColors`, `engineType`, `lWeight`, `cWeight`, `blackBias`, `vibrancyMode`, `vibrancyBoost`, `highlightThreshold`, `paletteReduction`, `hueLockAngle`, `shadowPoint`, `minVolume`, `speckleRescue`, `shadowClamp`, `ditherType`, `meshSize`, `trapSize`, `centroidStrategy`, `distanceMetric`, `quantizer`, `preprocessingIntensity`, etc.

#### `engine.setParams(obj)`

Batch version of `setParam`. Throws on any unrecognized key.

```javascript
engine.setParams({
    targetColors: 8,
    quantizer: "wu",
    centroidStrategy: "ROBUST_SALIENCY"
});
```

---

### Quantization

#### `engine.quantize(image, options?) → Palette`

Reads the image, runs preprocessing + posterization, returns a new Palette. The engine itself is not mutated. Options override current params for this call only.

```javascript
const palette = engine.quantize(image);
const palette = engine.quantize(image, { quantizer: "wu", targetColors: 12 });
```

Maps to: `BilateralFilter` (if preprocessing enabled) → `PosterizationEngine.posterize()`.

Dynamic archetype expressions are resolved here, since this is the first point where both the engine config and the image context are available.

---

### Separation

#### `engine.separate(image, palette, options?) → Result`

Maps every pixel to the nearest palette color. Returns a Result. The image and palette are not mutated.

```javascript
const result = engine.separate(image, palette);
const result = engine.separate(image, palette, { ditherType: "floyd-steinberg" });
```

Maps to: `SeparationEngine.mapPixelsToPaletteAsync()`.

## Palette API

Produced by `engine.quantize()`. The only mutable object in the pipeline.

### `palette.colors`

Read-only snapshot of current palette entries:

```javascript
palette.colors
// → [
//   { index: 0, l: 12, a: 0, b: 0, hex: "#1B1B1B", name: "Near Black", coverage: 0.18 },
//   { index: 1, l: 85, a: -5, b: 42, hex: "#FFBC4E", name: null, coverage: 0.12 },
//   ...
// ]
```

### `palette.find(query) → PaletteEntry`

Find a color by hex, index, or Lab proximity. **Throws if no match found.**

```javascript
palette.find("#000000")          // Nearest hex match (ΔE < 5)
palette.find(0)                  // By index
palette.find({ l: 50, a: 20 })  // Nearest Lab match
```

For replay on different images, the recipe author handles missing colors:

```javascript
// Option A: try/catch
try {
    palette.merge("#BBD900", "#FFBC4E");
} catch (e) {
    // color doesn't exist in this image's palette, skip
}

// Option B: check first
if (palette.has("#BBD900")) {
    palette.merge("#BBD900", "#FFBC4E");
}
```

### `palette.has(query) → boolean`

Check if a color exists without throwing.

### `palette.merge(source, target)`

Merge one color into another. Accepts hex, index, or Lab query for both arguments. Throws if either color not found.

```javascript
palette.merge("#BBD900", "#FFBC4E");  // Merge neon-green into gold
palette.merge(3, 1);                   // Merge index 3 into index 1
```

### `palette.remove(query)`

Remove a color. Throws if not found.

```javascript
palette.remove("#F0F0F0");  // Remove near-white
palette.remove(5);          // Remove by index
```

### `palette.add(color)`

Inject a color into the palette.

```javascript
palette.add({ l: 45, a: -12, b: 30, name: "Olive" });
palette.add("#8B4513");  // Hex → Lab conversion
```

### PaletteEntry methods

```javascript
const black = palette.find("#000000");
black.setWeight(2.5);            // Boost in SALIENCY centroid calculation
black.setName("Rich Black");     // Label for output layers
black.setLab(10, 0, 0);         // Nudge the centroid
black.lock();                    // Protect from merge/remove
```

## Result API

Produced by `engine.separate()`. Contains the separation output.

### `result.applyKnobs(settings)`

Apply mechanical knobs to the color index map. Can be called multiple times with different settings — each call operates on the current state. No re-separation required.

```javascript
result.applyKnobs({
    minVolume: 0.5,       // 0-5%  — ghost plate removal
    speckleRescue: 2,     // 0-10px — despeckle radius
    shadowClamp: 3.2      // 0-20% — minimum ink density
});

// Try different settings without re-separating
result.applyKnobs({ speckleRescue: 4 });
result.applyKnobs({ speckleRescue: 2 });  // back off
```

Maps to: `MechanicalKnobs.applyMinVolume()`, `.applySpeckleRescue()`, `.applyShadowClamp()`.

### `result.getMask(colorIndex) → Uint8Array`

Generate a binary mask for a single color (lazy, computed on demand).

```javascript
const blackMask = result.getMask(0);  // 255 where black, 0 elsewhere
```

### Result properties

```javascript
result.palette       // Palette snapshot at separation time
result.colorIndices  // Uint8Array — per-pixel palette index
result.width
result.height
result.metadata      // { archetype, quantizer, dna, processingMs }
```

## Dynamic Expressions in Archetypes

Archetype JSON parameters can contain expression strings that are evaluated at `quantize()` time against the image context.

```json
{
    "id": "salamander",
    "parameters": {
        "targetColors": "Math.min(image.dna.hue_entropy > 0.7 ? 10 : 7, channels + 2)",
        "speckleRescue": "image.width > 3000 ? 3 : 1",
        "lWeight": 1.1,
        "cWeight": "image.dna.c > 40 ? 2.5 : 2.0"
    }
}
```

**Evaluation context** (available variables):

| Variable | Type | Source |
|----------|------|--------|
| `image` | object | `{ width, height, bitDepth, dna, sectors, filename }` |
| `channels` | number | Current `targetColors` from slider or archetype |
| `Math` | object | Standard JS Math |

**Rules:**
- String values are evaluated as expressions; non-string values pass through unchanged
- Evaluation happens once at `quantize()` time (not per-pixel)
- Errors in expressions throw (fail loud — no silent fallbacks)
- Only pure expressions — no assignments, no function declarations, no side effects

**Implementation:** `new Function()` with a frozen scope containing only the context variables above. No `require`, no `import`, no globals.

## Record/Replay: Recipe Generation

The Navigator records every user action as an ordered list of API calls. On "Export Recipe," it emits a `.js` file.

### What Gets Recorded

| User Action | Recorded As |
|-------------|-------------|
| Select archetype | `engine.applyArchetype(name, overrides)` |
| Adjust slider | `engine.setParam(key, value)` |
| Quantize/re-quantize | `engine.quantize(image, options)` |
| Edit palette swatch | `palette.find(hex).setLab(l, a, b)` |
| Merge colors | `palette.merge(src, tgt)` |
| Remove color | `palette.remove(hex)` |
| Accept suggested color | `palette.add(color)` |
| Adjust knob | `result.applyKnobs({ key: value })` |
| Commit | `engine.separate(image, palette)` |

### What Does NOT Get Recorded

- Mouse movements, hover states, UI interactions
- Proxy renders (those are previews, not operations)
- Undo/redo history (the recipe is the final state, not the editing journey)

### Replay Semantics

A recipe replayed on a different image will likely fail at palette surgery because the quantized colors differ. This is expected — recipes are **starting points**, not pixel-exact reproductions. The recipe author adapts the surgery section for the new image, or wraps operations in try/catch for graceful degradation.

## Execution Hosts

Recipes run in any context that can provide pixels and consume results:

| Host | Source (Image) | Sink (Result) |
|------|----------------|---------------|
| **Navigator** | Photoshop active document | Fill+mask layers in PSD |
| **reveal-cli** | PNG/TIFF/PSD file | PNG plates, PSD, ORA |
| **reveal-batch** | Directory of images | Batch results + metrics |
| **Future GUI** | File open dialog | Any output format |

Each host constructs an Image, creates an Engine, runs the recipe, and handles output. The recipe is host-agnostic. Photoshop is just one source/sink — the engine doesn't know or care.

---

## Sandboxing

Recipes are JS functions that may be shared between users. The execution environment must be restricted:

- **Allowed:** `Math`, `JSON`, `Array`, `Object`, `String`, `Number` built-ins
- **Blocked:** `require`, `import`, `fetch`, `process`, `fs`, `eval`, `Function` (inside recipe scope)
- **Frozen:** The `image` object is deeply frozen (read-only)
- **No network, no filesystem, no globals** — the recipe can only talk to `engine`

Implementation: `vm.createContext()` in Node.js with a whitelist of globals. In UXP (Photoshop), recipes run in the plugin's already-sandboxed context.

## Open Questions

1. **Recipe versioning** — Should recipes declare a minimum engine version? What happens when the API changes?
2. **Chaining recipes** — Can a recipe call another recipe? (e.g., a "base" recipe for warm images that a specific recipe extends)
3. **Interactive vs. batch** — In Navigator, should the recipe pause at each phase for user review, or run straight through?
4. **Expression language scope** — Should dynamic archetype expressions have access to the palette (post-quantization), or only to image metadata (pre-quantization)?
5. **Lock semantics** — When a palette entry is locked, should `merge()` targeting it throw, or merge the other direction?
6. **Knob reset** — Should `result.applyKnobs()` be cumulative or reset to baseline each call? (Navigator currently supports both via baseline snapshots.)
